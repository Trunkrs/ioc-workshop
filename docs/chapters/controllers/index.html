2. Architect controllers

I choose a lazy person to do a hard job. Because a lazy person will find an easy way to do it.

- Bill Gates

Now let’s start with designing our architecture. As developers, we are lazy and should shy away from actually writing code. So we’re going to start at the top and work our way down the chain to make the most efficient progress in writing code. As we will only write what we need when we actually need it.

With this methodology in mind, let’s start at the outer bounds and work our way to the core of our application. In the previous part, we defined that the process starts with user interaction.

The controller#

Any entry point of our application will lead to a controller. Controllers are classes that host actions that can be called by an external actor. The responsibility of these actions is to assess the validity of the input and the delegation to a manager class that hosts the business logic for the action.

Let’s start by defining the actions an external actor can do with our vending machine:

class VendingMachineController {  public selectAndBuyDrink(    // The external actor supplies us only a number of the button clicked.    drink: number,  ): Promise<void> {    // TODO: Define action body  }}

We only have a single action on our controller. Even though it would be imaginable that the user would be able to cancel their transaction mid-selection, we didn’t define it as part of our process and thus it isn’t part of our implementation.

Now that we have defined our actions, we should start by writing down our expectations for our controller. These expectations come from our process definition and our business logic/constraints. By pre-defining these in tests, we will write our code in the most efficient way possible.

describe('VendingMachineController', () => {  let sut: VendingMachineController    beforeEach(() => {    sut = new VendingMachineController()  })
  describe('selectAndBuyDrink', () => {    it('should delegate the right drink type to the manager', () => {      // TODO: Test for our expectations    })  })})

Now to test our expectations we will need to stub the manager with the method we’re going to call. The manager has the responsibility of guarding the business rules and delegating to underlying repositories and services. Business rules should never be implemented outside of our business layer where the managers reside.

We don’t want to implement the manager yet because we’re still focusing on our controller. Let’s implement our stubbed manager.

class VendingMachineManager {  /**  * Executes a transaction to buy and dispense a drink.  * @param drinkNr The drink button number to dispense.  */  public buyDrink(drinkNr: number): Promise<void> {    // TODO: Implement logic  }}

Now that we have clearly defined our manager, we can add the manager as one of our controllers' dependencies. Some unchanged lines have been omitted for brevity.

class VendingMachineController {  public constructor(    private readonly manager: VendingMachineManager,  ) {}    [...]}

Let’s now implement our expectations fully while mocking the manager so that we can run the tests in a truly isolated fashion.

const createMockedManager = () => ({  buyDrink: jest.fn().mockResolvesValue(undefined)})
describe('VendingMachineController', () => {  let mockedManager = createMockedManager()  let sut: VendingMachineController    beforeEach(() => {    mockedManager = createMockedManager()    sut = new VendingMachineController(      mockedManager as unknown as VendingMachineManager,    )  })
  describe('selectAndBuyDrink', () => {    it('should delegate the right drink type to the manager', async () => {      const buttonNumber = 6            await sut.selectAndBuyDrink(buttonNumber)            expect(mockedManager.buyDrink).toBeCalledWith(buttonNumber)    })  })})

Now that we have written out our expectations, we can go ahead with implementing the controller itself. Some lines have been omitted for the sake of brevity.

class VendingMachineController {  [...]   public async selectAndBuyDrink(    // The external actor supplies us only a number of the button clicked.    drink: number,  ): Promise<void> {    await this.manager.buyDrink(drink)  }}

And we are done! We have written a controller and established the basic outline, also called stubbing, of our manager that will handle the business rules of our vending machine.
